# Shopify CRO Opportunity Engine

[![CI](https://github.com/RickyVishwakarma/shopify-CRO/actions/workflows/ci.yml/badge.svg)](https://github.com/RickyVishwakarma/shopify-CRO/actions/workflows/ci.yml)

> Paste a Shopify store URL, get a **prioritized, evidence-backed Conversion Rate Optimization audit** — generated by an LLM working from the store's *real* catalog and page content, not generic best-practice advice.

This is an internal-consultant-style tool: the kind of thing a CRO agency would run before a sales call to walk in already knowing where a store is leaving money on the table.

```
  https://gymshark.com  ─►  [ scrape ]  ─►  [ extract evidence ]  ─►  [ LLM ]  ─►  ranked audit
```

**Example output (abridged):**

> **Opportunity #1 — Surface social proof on product pages** · Priority 4.0
> **Problem:** Product pages show no review count above the fold, while the homepage hero claims "10M+ customers."
> **Evidence:** `PDP: Vital Seamless Leggings` → `product.reviews` field is empty; homepage `.hero__subtitle` = "Trusted by 10M+ athletes".
> **Impact** 4/5 · **Confidence** 0.8 · **Effort** 2/5
> **Experiment:** Add review stars + count near the Add-to-Cart button. *Hypothesis:* social proof at decision point lifts add-to-cart rate. *Metric:* PDP→cart conversion. *Expected:* +3–6%.

## What it does

1. **Validates** the submitted URL (and refuses private/internal hosts — SSRF-safe).
2. **Scrapes** the store: homepage, collections, and a sample of product pages.
3. **Extracts** structured evidence — products, collections, hero, CTAs, trust badges, pricing, reviews, shipping/returns copy, FAQ, navigation, images.
4. **Analyzes** the evidence with an LLM that is required to return **strict JSON only** and to **cite the exact page element behind every recommendation**.
5. **Ranks** opportunities by a transparent priority score.
6. **Presents** a dashboard: executive summary, impact/effort chart, and a filterable, searchable list of opportunities with expandable reasoning and one-click experiment briefs.

### What makes the output *good* (not generic)

Three things separate this from a tool that just tells every store "add reviews and improve your CTA":

- **Evidence-backed recommendations.** Every opportunity links to the specific scraped element or text that triggered it. If we can't point at evidence, we don't make the claim.
- **One-click experiment briefs.** Each recommendation ships with a ready-to-run A/B test: hypothesis, metric, expected impact, effort, and implementation note.
- **Executive summary.** A plain-language TL;DR written for a non-technical marketing manager, on top of the detailed engineer-facing audit.

## Architecture

```
 Browser (Next.js client)
   │  POST /api/audits { url }
   ▼
 Route handler  ── orchestrator.ts ────────────────────────────────────┐
   │                                                                    │
   ├─► URL validation + SSRF guard ──────────► reject bad/private hosts │
   │                                                                    │
   ├─► Cache lookup (normalized URL) ─────────► hit? return cached      │
   │                                                                    │
   ├─► Scraper  (async, parallel, fault-tolerant)                       │
   │     ├─ Shopify JSON:  /products.json, /collections.json            │
   │     ├─ Homepage HTML  ─► cheerio parse (hero, CTAs, trust, nav…)    │
   │     └─ Sample PDP HTML ─► cheerio parse (reviews, shipping, FAQ)    │
   │     └─ each call: timeout + retry + partial-failure tolerance      │
   │                                                                    │
   ├─► Normalize ─────────────────────────────► StoreEvidence (typed,   │
   │                                              token-bounded)         │
   ├─► LLM provider  (dependency-injected interface)                    │
   │     ├─ ClaudeProvider   strict JSON · zod validate · repair retry  │
   │     ├─ GeminiProvider   free tier · forced JSON · retry + backoff  │
   │     └─ TemplateProvider deterministic fallback — runs with no key  │
   │                                                                    │
   ├─► Grounding check ───────────────────────► flag any cited evidence │
   │                                              not found in the scrape│
   ├─► Scoring  (Impact × Confidence ÷ Effort, recomputed server-side)  │
   │                                                                    │
   └─► Persist to cache ──────────────────────► return Audit ───────────┘
```

The whole pipeline lives in one Next.js app — there is no separate backend service. See the decisions section for why.

## Key engineering decisions (and the tradeoffs)

This section is the point. Every choice below was made to fit a **small, real, shippable** tool — and to be honest about where the design's edges are.

### 1. Shopify's public JSON endpoints instead of a headless browser

Shopify stores expose `/products.json` and `/collections.json` as **public, structured, paginated** endpoints. We use them for catalog data and a lightweight HTML fetch + [cheerio](https://cheerio.js.org/) parse only for page *chrome* (hero, CTAs, trust badges, review widgets).

- **Why:** A headless browser (Playwright/Puppeteer) for a tool that reads three page types is over-engineering — it adds a ~400 MB Chromium dependency, slow cold starts, and anti-bot fragility, in exchange for nothing the JSON endpoints don't already give us cleanly.
- **Tradeoff / edge:** Themes that render entirely client-side, or stores that disable the JSON endpoints, won't fully parse. At real scale this scraper would graduate to a job queue + worker + headless browser. For this tool's scope, plain HTTP is the right call — and recognizing that is itself the engineering judgment.

### 2. One Next.js app, not a Next.js frontend + Python/FastAPI backend

- **Why:** A 4–6 hour, single-purpose tool doesn't justify two services, two deployments, and the CORS/contract glue between them. Next.js route handlers do async orchestration perfectly well. Fewer moving parts means more time spent on what's actually graded — the *quality of the analysis* — and a reviewer can clone and run it with one command.
- **Tradeoff:** Heavy CPU-bound scraping shares the request lifecycle. If this grew into a product, the scrape+analyze step would move to a background worker with a job ID the client polls. The current design streams progress over SSE so the synchronous flow still feels responsive.

### 3. The LLM sits behind a provider interface, with a deterministic fallback

`LLMProvider` is an interface with three implementations, tried in order:
- `ClaudeProvider` — analysis via the Anthropic API, using structured-output (JSON-schema) mode.
- `GeminiProvider` — analysis via Google Gemini (free tier), forcing JSON output. A free way to run the real model-driven path.
- `TemplateProvider` — a rule-based generator that produces a valid (if shallower) audit **with no API key at all**.

At runtime, `analyzeStore` uses the first configured provider (Claude if `ANTHROPIC_API_KEY` is set, else Gemini if `GEMINI_API_KEY` is set), and falls back to the next — and ultimately to the template — on any failure.

- **Why:** It makes the LLM a swappable dependency (trivially mockable in tests), lets the model path run for free via Gemini, and means **the app runs and demos even with no key configured** — it degrades, it doesn't die. That is the "never depend on the model blindly; fail gracefully" principle made concrete.
- **Tradeoff:** The template fallback's output is rule-based and less nuanced. It's a safety net and a test seam, not a replacement for the model.

### 4. We never trust the model's output blindly

Three guardrails sit between the LLM and the user:
1. **Schema validation** — the response is constrained to a JSON schema *and* re-validated with [zod](https://zod.dev/) server-side. One automatic "repair" retry on malformed output, then graceful degradation.
2. **Server-side scoring** — `priorityScore` is **recomputed from impact/confidence/effort in our code**, never read from the model. The model can't fudge the ranking.
3. **Grounding check** — every `evidence.excerpt` the model cites is checked against the actual scraped data; anything it can't be matched to is flagged rather than shown as fact. This catches hallucination programmatically.

### 5. Current stable Next.js (16), not the brief's "15"

The brief named Next.js 15; the current stable is 16, which the scaffold installed. Both use the same App Router model this project relies on. Pinning backwards to 15 would buy nothing and forgo current-stable fixes — so we're on 16 and noting it here rather than silently diverging.

### Deliberately out of scope (v1)

Scope discipline is a feature. The following are intentionally *not* built, and listed under [Future work](#future-work): user accounts/auth, a historical-audit database, JS-rendered theme scraping, and PDF export (JSON export ships instead — cheaper and just as useful for a v1).

## How a recommendation is produced

```
scraped data ─► normalize to StoreEvidence ─► prompt (system + evidence + JSON schema)
   ─► Claude (structured output) ─► zod validate ─► grounding check
   ─► recompute priority ─► sort ─► render
```

**Priority score:**

```
priority = (impact × confidence) ÷ effort
           impact:     1–5   (business upside if fixed)
           confidence: 0–1   (how sure we are it's a real problem)
           effort:     1–5   (implementation cost; floored at 1)
```

High-impact, high-confidence, low-effort opportunities rise to the top — the same mental model a good consultant uses to decide what to do first.

**Prompt strategy (summary):** the system prompt casts the model as a senior Shopify CRO consultant with hard rules — *output JSON only; every opportunity must cite supplied evidence; never invent products, prices, or copy; if data is missing, say so as a "missing" evidence type rather than guessing.* Full detail in [`lib/prompts/`](lib/prompts).

## Running locally

```bash
npm install
cp .env.example .env.local     # add your Anthropic key (optional — see below)
npm run dev                    # http://localhost:3000
```

Open the app, paste a Shopify store URL (e.g. `https://gymshark.com`), and run an audit.

> **No API key?** The app still runs — it falls back to the deterministic `TemplateProvider` and clearly labels the result as a fallback audit. Set `ANTHROPIC_API_KEY` or a free `GEMINI_API_KEY` (see below) for full LLM-quality output.

## Environment variables

| Variable | Required | Description |
|---|---|---|
| `ANTHROPIC_API_KEY` | No* | Anthropic API key (preferred provider). Get one at [platform.claude.com](https://platform.claude.com). |
| `LLM_MODEL` | No | Claude model id (default `claude-opus-4-8`; e.g. `claude-haiku-4-5` to cut cost). |
| `GEMINI_API_KEY` | No* | Google Gemini key — a **free** way to run the real model path. Get one at [aistudio.google.com/apikey](https://aistudio.google.com/apikey). |
| `GEMINI_MODEL` | No | Gemini model id (default `gemini-2.5-flash`). |
| `MAX_PRODUCTS` | No | Cap on products sampled per audit (controls token cost/latency). |

\* Neither key is required to run — with no key set, the app uses the deterministic fallback. Set **either** key for real LLM-quality audits.

## Why it's built this way

The brief asked for product judgment, code quality, and thoughtful use of the model — not pixel perfection. So the design optimizes for exactly that: the simplest architecture that fully solves the problem, every external dependency wrapped to fail gracefully, the model's output never trusted blindly, and one genuinely useful differentiator (evidence-backed recommendations) executed well. Where a decision trades something away, that tradeoff is named here rather than hidden.
